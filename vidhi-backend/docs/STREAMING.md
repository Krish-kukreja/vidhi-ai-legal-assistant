## Overview

VIDHI supports real-time streaming for both text and voice interactions, providing a modern, responsive user experience similar to ChatGPT.

---

## Text Streaming (SSE)

### Endpoint
```
POST /api/v1/stream/chat
```

### Request
```json
{
  "text": "What is Section 438 CrPC?",
  "session_id": "user-session-123",
  "language": "english"
}
```

### Response (Server-Sent Events)
```
event: token
data: {"type":"token","content":"Section","index":0}

event: token
data: {"type":"token","content":" 438","index":1}

event: metadata
data: {"type":"metadata","confidence":0.85,"citations":["Section 438 CrPC"]}

event: done
data: {"type":"done","total_tokens":150,"duration_ms":2500}
```

### Event Types

#### 1. Token Event
```json
{
  "type": "token",
  "content": "Section",
  "index": 0
}
```
Sent for each token generated by the LLM.

#### 2. Metadata Event
```json
{
  "type": "metadata",
  "confidence": 0.85,
  "citations": ["Section 438 CrPC"]
}
```
Sent after all tokens, contains confidence score and citations.

#### 3. Done Event
```json
{
  "type": "done",
  "total_tokens": 150,
  "duration_ms": 2500
}
```
Sent when streaming is complete.

#### 4. Error Event
```json
{
  "type": "error",
  "message": "LLM service unavailable"
}
```
Sent if an error occurs during streaming.

## Voice Streaming (WebSocket)

### Endpoint
```
WS /api/v1/ws/voice
```

### Connection Flow

#### 1. Connect to WebSocket
```javascript
const ws = new WebSocket('ws://localhost:8000/api/v1/ws/voice');
```

#### 2. Initialize Session
```json
{
  "type": "start_session",
  "session_id": "user-session-123",
  "language": "english",
  "language_code": "en-IN"
}
```

#### 3. Send Audio Chunks
```json
{
  "type": "audio_chunk",
  "data": "base64_encoded_audio",
  "format": "webm"
}
```

#### 4. Signal Audio Complete
```json
{
  "type": "audio_complete",
  "format": "webm"
}
```

#### 5. Receive Events
- `session_started`: Session initialized
- `chunk_received`: Audio chunk acknowledged
- `transcription`: Speech-to-text result
- `llm_token`: LLM response token
- `audio_url`: TTS audio URL
- `status`: Processing status
- `done`: Processing complete
- `error`: Error occurred

#### 6. End Session
```json
{
  "type": "end_session"
}
```

## Frontend Integration

### Text Streaming

```typescript
import { streamChat } from '@/api/apiClient';

async function handleChat(text: string) {
  let fullResponse = '';
  
  for await (const event of streamChat(text, sessionId, language)) {
    if (event.type === 'token') {
      fullResponse += event.content;
      updateUI(fullResponse);
    } else if (event.type === 'metadata') {
      showMetadata(event.confidence, event.citations);
    } else if (event.type === 'done') {
      console.log('Streaming complete');
    } else if (event.type === 'error') {
      showError(event.message);
    }
  }
}
```

### Voice Streaming

```typescript
import { VoiceStreamingService, AudioRecorder } from '@/services/voiceStreamingService';

// Initialize service
const service = new VoiceStreamingService({
  sessionId: 'user-session-123',
  language: 'english',
  languageCode: 'en-IN',
  onTranscription: (text) => console.log('Transcript:', text),
  onLLMToken: (content) => console.log('Token:', content),
  onAudioUrl: (url) => playAudio(url),
  onDone: (data) => console.log('Done:', data)
});

// Connect
await service.connect();

// Record audio
const recorder = new AudioRecorder();
await recorder.startRecording();

// ... user speaks ...

// Stop and send
const audioBlob = await recorder.stopRecording();
await service.sendAudioChunk(audioBlob, 'webm');
service.sendAudioComplete('webm');

// Disconnect
service.disconnect();
```

## Performance

### Text Streaming
- **First token latency**: < 500ms
- **Token throughput**: 10-20 tokens/second
- **Memory usage**: Minimal (async generators)

### Voice Streaming
- **WebSocket connection**: < 100ms
- **Audio upload**: < 500ms
- **Transcription**: 1-3s (batch mode)
- **LLM first token**: < 500ms
- **TTS generation**: 500ms-1s
- **Total end-to-end**: 3-7s

---

## Error Handling

### Text Streaming Errors
- **Connection timeout**: Retry with exponential backoff
- **LLM service unavailable**: Fall back to regular request
- **Network error**: Show error message, allow retry

### Voice Streaming Errors
- **Microphone access denied**: Show permission instructions
- **WebSocket connection failed**: Retry up to 3 times
- **Transcription failed**: Show error, allow re-recording
- **LLM error**: Show error message
- **TTS failed**: Show text response only

---

## Security

### Authentication
- Optional JWT token in `Authorization` header
- Guest access allowed (no auth required)
- Session isolation enforced

### Input Sanitization
- All text inputs sanitized (XSS prevention)
- Session IDs validated
- Audio data validated (format, size)

### Rate Limiting
- 100 requests per minute per user
- WebSocket connections limited per IP
- DoS protection active

## Troubleshooting

### Text Streaming Issues

**Problem**: No tokens received  
**Solution**: Check LLM service status, verify network connection

**Problem**: Slow streaming  
**Solution**: Check network latency, verify AWS Bedrock region

**Problem**: Connection timeout  
**Solution**: Increase `STREAMING_TIMEOUT`, check firewall

### Voice Streaming Issues

**Problem**: Microphone not working  
**Solution**: Check browser permissions, verify HTTPS connection

**Problem**: Transcription failed  
**Solution**: Check audio format, verify AWS Transcribe configuration

**Problem**: No audio playback  
**Solution**: Check TTS service, verify audio URL accessibility

**Problem**: High latency  
**Solution**: Optimize audio encoding, use streaming transcription

## Best Practices

### Text Streaming
1. Always handle all event types (token, metadata, done, error)
2. Implement cancellation for long responses
3. Show progress indicator during streaming
4. Fall back to regular requests if streaming fails
5. Cache responses for repeated queries

### Voice Streaming
1. Request microphone permission early
2. Show clear recording indicator
3. Provide visual feedback (waveform)
4. Handle reconnection gracefully
5. Clean up resources on disconnect
6. Show status messages at each stage
7. Allow cancellation at any time
